# 建設DXサポートシステム構築 変更管理手順書

**プロジェクト名**: CDCP (Construction Digital Control Platform) 構築プロジェクト
**バージョン**: 1.0
**作成日**: 2026-02-15
**適用開始**: Phase 1開始時

---

## 1. 目的

本手順書は、建設DXサポートシステム構築プロジェクトにおいて、プロジェクトスコープ・要件・設計・計画などの変更要求を、体系的に管理し、プロジェクト目標への影響を最小化することを目的とします。

---

## 2. 適用範囲

### 2.1 対象変更

本手順書が適用される変更:

| 変更種類 | 対象 | 例 |
|---------|------|-----|
| **スコープ変更** | 機能追加・削除・修正 | 新しい工程管理機能追加、指摘管理削除 |
| **要件変更** | 非機能要件、性能要件 | 稼働率目標引き上げ、接続数変更 |
| **設計変更** | システム構成、DB設計 | UI/UX改善、テーブル構造変更 |
| **スケジュール変更** | 納期、マイルストーン | Phase 1完了予定を1ヶ月延期 |
| **予算変更** | コスト増減、予算配分 | ベンダー費用追加、ライセンス数変更 |
| **ステークホルダー変更** | 関係者追加・削除 | 新しい工事部門参画、協力会社追加 |
| **優先度変更** | 機能実装順序 | 写真管理機能の優先度引き上げ |
| **リスク対応変更** | リスク対策内容 | ベンダー複数社手配→1社に集約 |

### 2.2 対象外

以下は変更管理外(通常の業務処理):
- 日常的なバグ修正(軽微なUI改善など)
- 既存要件の解釈説明・澄み分け
- ドキュメント誤字修正
- 参考資料の追加

## 3. 変更管理プロセス

### 3.1 全体フロー

```
①変更要求書提出
    │
    ↓
②初期スクリーニング
    │
    ├─ 軽微 → ④迅速承認
    ├─ 通常 → ③詳細分析へ
    └─ 重大 → ③+経営会議
    │
    ↓
③影響分析(詳細)
    ├─ スコープ影響
    ├─ 予算影響
    ├─ スケジュール影響
    ├─ リスク影響
    ├─ 技術的影響
    └─ ステークホルダー影響
    │
    ↓
④変更評価会議
    ├─ 承認 → ⑤実装
    ├─ 条件付承認 → 修正後⑤へ
    ├─ 保留 → 次フェーズ以降検討
    └─ 却下 → ⑥却下報告
    │
    ↓
⑤変更実装
    ├─ 要件書修正
    ├─ 設計修正
    ├─ 開発実装
    ├─ テスト実施
    └─ 承認・記録
    │
    ↓
⑥変更完了・報告
```

### 3.2 変更分類

変更は以下の3段階に分類されます:

#### 3.2.1 **軽微変更(Level 1)**

**判定基準**:
- スコープ外の小規模修正
- 予算影響: ±50万円以内
- スケジュール影響: ±3営業日以内
- 影響範囲: 1つの機能/モジュール以内
- リスク: 低

**承認者**: プロジェクトマネージャー
**承認時間**: 1営業日以内
**記録**: 軽微変更ログ(集計して月次報告)

**例**:
- 画面表示項目の順序変更
- エラーメッセージ文言修正
- 既存UIの小規模改善
- テーブルカラム追加(DB影響なし)

#### 3.2.2 **通常変更(Level 2)**

**判定基準**:
- スコープ内の機能修正/改善
- 予算影響: ±50万円〜±200万円
- スケジュール影響: ±1週間〜±1ヶ月
- 影響範囲: 複数機能/モジュール
- リスク: 中程度

**承認者**: プロジェクト委員会(PM、業務責任者、IT責任者)
**承認時間**: 3-5営業日
**記録**: 変更要求書、影響分析書

**例**:
- 工程管理画面の大幅改善
- API仕様の部分修正
- 性能要件の調整(99% → 99.5%)
- 協力会社マスタへのフィールド追加

#### 3.2.3 **重大変更(Level 3)**

**判定基準**:
- スコープ大幅変更(新機能追加など)
- 予算影響: ±200万円以上
- スケジュール影響: 1ヶ月以上
- 影響範囲: システム全体に波及
- リスク: 高

**承認者**: 経営会議(CEO, CFO, 工事部門長, IT責任者)
**承認時間**: 1週間程度
**記録**: 変更要求書、詳細影響分析書、提案資料

**例**:
- 新しい管理機能全体の追加(維持管理機能など)
- Phase 1の1ヶ月延期
- ベンダー変更
- 予算3,500万円→5,000万円へ増額

## 4. 変更要求手続き

### 4.1 Step 1: 変更要求書作成・提出

#### 4.1.1 誰が提出できるか

以下のステークホルダーが変更要求書を提出できます:

- プロジェクト委員会メンバー
- 開発ベンダーのプロジェクトマネージャー
- IT部門責任者
- 工事部門長・現場監督(業務要件関連)
- 本社管理部門(予算・原価要件関連)

### 4.1.2 変更要求書テンプレート

**必須項目**:

```
【変更要求書】

1. 基本情報
   要求日: ________年__月__日
   要求者: ____________________
   要求者部門: ____________________
   優先度: □高 □中 □低

2. 変更概要
   変更タイトル: ____________________
   変更内容(詳細): ____________________

3. 変更背景・理由
   (具体的な理由を記載)
   ____________________

4. 現在の状態(As-Is)
   ____________________

5. 期待される状態(To-Be)
   ____________________

6. 変更の影響範囲(初期推定)
   □スコープ □予算 □スケジュール □リスク □技術 □その他

7. 提案者からの提案
   (予算・スケジュール案があれば記載)
   ____________________

8. 添付資料
   □影響図 □試算データ □提案書 □その他___________
```

### 4.1.3 提出方法・期限

**提出方法**:
- 変更要求書をPM宛にメール送付
- または、プロジェクト管理ツールで登録

**提出期限**:
- 緊急でない限り、月1回の「変更管理レビュー会議」(毎月第2火曜14時)の1週間前まで
- 緊急変更は、その都度対応可能(経営会議判定)

### 4.2 Step 2: 初期スクリーニング

#### 4.2.1 実施者・時期

**実施者**: プロジェクトマネージャー
**時期**: 要求書提出後2営業日以内

#### 4.2.2 スクリーニング項目

| 項目 | 判定基準 | 判定方法 |
|-----|---------|---------|
| 完全性 | 必須項目が全て記載されているか | 要求書チェック |
| 明確性 | 変更内容が明確か | PM判定、不明な点は提出者に確認 |
| スコープ適合性 | 本プロジェクト内の変更か | 企画書・要件書との照合 |
| 実現可能性 | 技術的に実現可能か | 初期判定、ベンダーに相談 |
| 優先度分類 | Level 1, 2, 3のどれか | 影響度判定表で分類 |

#### 4.2.3 スクリーニング結果

結果は変更要求者に通知:

| 結果 | 処理 | 連絡先 |
|-----|------|--------|
| **却下** | 範囲外・不適切 → 要求者へ却下理由説明 | メール、会議 |
| **Level 1承認** | PM判定で即承認 → 変更管理ログ記録 | メール通知 |
| **Level 2進行** | 詳細分析へ → 分析着手通知 | メール+プロジェクト会議 |
| **Level 3進行** | 経営会議判定へ → 経営会議スケジュール調整 | 電話+メール |

### 4.3 Step 3: 影響分析(Level 2, 3対象)

#### 4.3.1 分析項目

**A. スコープ影響分析**

| 分析項目 | 内容 | 記載方法 |
|---------|------|---------|
| スコープ拡大/縮小 | 新機能追加か既存機能修正か | ○か×で判定 |
| 影響を受ける機能 | 関連機能リスト | 機能ID、機能名 |
| 他フェーズへの影響 | Phase 2, 3への波及 | 記述 |
| 利用者への影響 | 操作方法変更など | 記述 |

**B. 予算影響分析**

| 分析項目 | 内容 | 記載方法 |
|---------|------|---------|
| 追加費用 | 新規購入、ベンダー追加 | 金額 |
| 削減可能費用 | 既存予算から削減可能 | 金額 |
| 予備費からの充当 | 予備費(全体予算10%)から配分 | 金額 |
| 総予算への影響 | 初期投資額/年間運用費への影響 | % |

**例計算**:
```
変更内容: AI画像品質チェック機能追加(Phase 2)
追加費用: 250万円
- AI学習用データセット: 80万円
- 開発工数(200h): 170万円

削減可能費用: 0円
予備費からの充当: 250万円
総予算(9,920万円)への影響: +2.5% → 10,170万円
```

**C. スケジュール影響分析**

| 分析項目 | 内容 | 記載方法 |
|---------|------|---------|
| 追加開発工数 | 見積り工数 | 人月単位 |
| 追加テスト期間 | テスト期間延長 | 日数 |
| 他タスクへの影響 | 他タスク延期の有無 | 機能名、延期日数 |
| Phase全体への影響 | Phase開始/終了日への影響 | 日数 |
| クリティカルパス影響 | 関鍵タスク遅延 | ○か× |

**例計算**:
```
変更内容: ガントチャート表示性能改善(Phase 1内)
追加工数: 100時間 = 2.5人月(外部ベンダー)
Phase 1現在の開発スケジュール: 2ヶ月目(開発2ヶ月のうち)
残工数: 400時間
追加100時間 → 全体1,600時間(計画) + 100時間
→ 完了予定: 2ヶ月目開始+15日延期 → 2.5ヶ月目中旬
クリティカルパス影響: ○(テスト期間圧迫)
```

**D. リスク影響分析**

| 分析項目 | 内容 | 記載方法 |
|---------|------|---------|
| 新規リスク | 変更に伴う新しいリスク | リスク名、確率、影響 |
| 既存リスク増加 | 既存リスク顕在化確率の上昇 | リスク名、変更後の確率 |
| 技術リスク | 技術的な不確実性 | 記述 |
| ベンダー依存度 | ベンダー依存度の上昇 | % |

**例**:
```
変更内容: 会計システム連携機能前倒し(Phase 1内)
新規リスク:
- R1: 既存会計システムのデータ仕様が不明確
  確率: 中, 影響度: 中 → 対策: ベンダーサーベイ実施
- R2: 連携テストが十分に実施できない
  確率: 中, 影響度: 中 → 対策: テスト期間確保
```

**E. 技術的影響分析**

| 分析項目 | 内容 | 記載方法 |
|---------|------|---------|
| システムアーキテクチャ影響 | DB変更、API追加など | 図解 |
| 既存機能への互換性 | 既存機能が動作するか | ○か× |
| パフォーマンス影響 | 応答時間、スループット | 定量値 |
| セキュリティ影響 | 権限管理、データ保護 | 記述 |

**F. ステークホルダー影響分析**

| ステークホルダー | 影響 | 対応方法 |
|-------------------|------|---------|
| 現場監督 | 操作方法変更 → 教育・マニュアル更新 | 対策リスト |
| 協力会社 | システム利用範囲拡大 | 対策リスト |
| IT部門 | 運用工数増加 → 体制調整 | 対策リスト |
| 経営層 | 予算追加 → 経営会議報告 | 対策リスト |

#### 4.3.2 影響分析書作成

**実施者**: PM + ベンダーPM + 業務責任者
**期間**: 3-5営業日
**成果物**: 「変更要求書 - 影響分析書」(変更要求書に添付)

**影響分析書テンプレート**:

### 4.4 Step 4: 変更評価会議

#### 4.4.1 会議体

**Level 2変更評価会議**
- **実施者**: プロジェクト委員会(PM、業務責任者、IT責任者、ベンダーPM)
- **頻度**: 毎月第2火曜14:00-15:30
- **議事**: 当月提出のLevel 2変更要求書を一括レビュー
- **意思決定**: 3/4以上の賛成で承認

**Level 3変更評価会議(経営会議)**
- **実施者**: 経営会議(CEO, CFO, 工事部門長, IT部門責任者, PM)
- **頻度**: 月1回(第4火曜)
- **議事**: Level 3変更要求書、四半期レビューと合併
- **意思決定**: 4/5以上の賛成で承認

#### 4.4.2 評価基準

| 判定 | 基準 | 処理 |
|-----|------|------|
| **承認(Approve)** | 変更が有益で、予算・スケジュール対応可能 | 実装フェーズへ |
| **条件付承認(Conditional)** | 一部修正で承認可能(例:予算調整、スケジュール調整) | 要求者に修正依頼、再提案 |
| **保留(Defer)** | 次フェーズ以降で検討が適切 | 要求書を次Phase計画に組み込む |
| **却下(Reject)** | 変更の有益性が低い、または実現困難 | 却下理由を記載、アーカイブ |

#### 4.4.3 意思決定基準

**自動承認条件**(会議不要):
- 予算影響 ±50万円以内 AND
- スケジュール影響 ±3営業日以内 AND
- リスク影響 低 AND
- 技術的障害なし

**会議必須条件**:
- 予算影響 ±50万円超 OR
- スケジュール影響 ±1週間超 OR
- リスク影響 中以上 OR
- ステークホルダー影響 大 OR
- 経営判断が必要

### 4.5 Step 5: 変更実装

#### 4.5.1 実装計画書作成

**承認後**、以下を実施:

1. **変更実装計画書の作成**
   - 対象: 承認されたLevel 2, 3変更
   - 作成者: ベンダーPM
   - 内容: 詳細な実装手順、工数配分、リスク対応

2. **関係者への周知**
   - PM → プロジェクト委員会、ベンダー、ユーザー代表へ通知
   - メール + プロジェクト会議で説明

3. **ドキュメント更新計画**
   - 要件書、設計書、テスト計画書などの更新スケジュール
   - 成果物納期への影響確認

#### 4.5.2 実装フェーズ

**フェーズ**:
1. 詳細設計修正
2. 開発実装
3. 単体テスト
4. 統合テスト
5. 承認・記録

**各フェーズの品質確認**:
- 設計レビュー: PM, ベンダーPM, IT責任者
- コード レビュー: 開発チーム
- テスト検証: QAチーム + 業務責任者
- 最終承認: PM

---

### 4.6 Step 6: 変更完了・報告

#### 4.6.1 変更完了基準

変更が「完了」と認められるには:

- [ ] 詳細設計書が修正され、PMレビュー済み
- [ ] 開発実装が完了し、コードレビュー通過
- [ ] 統合テストで合格(テスト成績書サイン)
- [ ] 要件書/設計書/マニュアル更新完了
- [ ] ステークホルダーが内容確認・承認
- [ ] 変更管理ログが記録済み

#### 4.6.2 変更完了報告

**報告内容**:
- 変更ID、変更タイトル
- 実装概要、成果物
- 実績工数(計画vs実績)
- 実績スケジュール(計画vs実績)
- 予算使用額(計画vs実績)
- 発見された新規リスク
- 次フェーズへの懸案

**報告先**:
- プロジェクト委員会(月次)
- 経営会議(四半期レビュー)
- ステークホルダー(関連者)

#### 4.6.3 変更管理ログ更新

**記録内容**:

| 項目 | 内容 |
|-----|------|
| 変更ID | CR-YYYYMM-001形式 |
| 変更タイトル | 簡潔に記載 |
| 変更者 | 提出者名 |
| 提出日 | 要求書提出日 |
| 承認日 | 承認決定日 |
| 承認者 | PM or 経営会議 |
| 実装予定日 | 開始日-完了日 |
| 実装完了日 | 実績完了日 |
| 予算影響 | 計画額-実績額 |
| スケジュール影響 | 計画期間-実績期間 |
| 状態 | 承認/条件付/保留/却下/完了 |

## 5. 変更の評価基準

### 5.1 予算評価基準

| 影響度 | 予算変更幅 | 対応 | 承認者 |
|--------|----------|------|--------|
| **軽微** | ±50万円以内 | 予備費(全体10%)から配分 | PM判定 |
| **中** | ±50万円〜±200万円 | 該当Phase予算を調整 | プロジェクト委員会 |
| **大** | ±200万円以上 | 経営会議で新規予算承認 | 経営会議 |

**予備費活用ルール**:
- 各Phase予算に予備費10%を設定
- 軽微・通常変更はこれから充当
- 予備費を超える場合は新規予算要求
- 予備費残額は次Phaseに繰越

### 5.2 スケジュール評価基準

| 影響度 | スケジュール変更 | 対応 | 承認者 |
|--------|-----------------|------|--------|
| **軽微** | ±3営業日以内 | 該当スプリント内で調整 | PM判定 |
| **中** | ±1週間〜±1ヶ月 | Phase内で調整、他スプリント影響検討 | プロジェクト委員会 |
| **大** | 1ヶ月以上 | Phase完了予定日変更、Go/No-Go判定 | 経営会議 |

**スケジュール調整方法**:
1. 他タスク優先度の見直し → 並行タスク削減
2. 追加リソース投入 → ベンダーリソース増員
3. 機能削減 → Phase次送り
4. Phase期間延長 → 経営会議承認

### 5.3 リスク評価基準

| リスク種別 | 評価項目 | 判定 | 対応 |
|-----------|---------|-----|------|
| **技術的リスク** | 実現可能性 | 困難 | ベンダー技術サーベイ、スコープ縮小検討 |
| **運用リスク** | IT部門負荷 | 過大 | リソース追加、外部委託検討 |
| **品質リスク** | テスト期間不足 | 高 | テスト期間確保、自動テスト導入 |
| **セキュリティリスク** | 脆弱性導入 | あり | セキュリティ診断追加 |

### 5.4 ステークホルダー影響評価

| ステークホルダー | 評価項目 | 影響大 | 対応 |
|------------------|---------|--------|------|
| **現場監督** | 操作方法変更の大きさ | 大幅変更 | 研修・マニュアル事前準備 |
| **協力会社** | 参加負荷増加 | 大幅 | インセンティブ検討、段階展開 |
| **IT部門** | 運用工数増 | 週10h超過 | 外部ヘルプデスク検討 |
| **経営層** | 投資利益率低下 | ROI 低下 | 代替案検討、効果測定 |

## 7. 特殊なシナリオ

### 7.1 緊急変更(Critical Change)

**判定基準**:
- システムダウン対応
- セキュリティ侵害対応
- 契約上の必須要件発動
- 法令対応(コンプライアンス)

**対応フロー**(通常と異なる):

```
①緊急発報 → ②PM即時判定
    ↓
③応急対応実施(即座)
    ↓
④経営会議に報告(同日)
    ↓
⑤変更要求書事後作成
    ↓
⑥影響分析・承認
```

**例**: セキュリティ脆弱性発見 → 即座にパッチ適用 → 翌営業日に変更要求書作成

### 7.2 大規模スコープ変更

**判定**: Level 3相当で、複数の機能・システム影響

**対応**:
1. 詳細なビジネス要件ヒアリング(1週間)
2. ベンダーテクニカルサーベイ(1週間)
3. 複合影響分析(複数ドメイン: 予算、スケ、リスク、技術)
4. 経営戦略会議で「Phase 内実施 vs 新プロジェクト化」判定
5. 必要に応じてプロジェクト憲章修正

### 7.3 複数変更の関連管理

**場面**: 複数の関連変更が同時期に提出された

**対応**:
1. 変更同士の依存関係・相互影響を分析
2. 「変更パッケージ」として一括管理
3. 共通の影響分析ドキュメント作成
4. 統合的な予算・スケジュール評価

**例**: 「工程管理UI改善」「工程データモデル修正」「工程APIリファクタリング」の3つの関連変更
