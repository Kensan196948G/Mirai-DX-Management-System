# 建設DXサポートシステム構築 コミュニケーション計画書

**プロジェクト名**: CDCP (Construction Digital Control Platform) 構築プロジェクト
**バージョン**: 1.0
**作成日**: 2026-02-15
**適用開始**: Phase 1開始時

---

## 1. 目的

本計画書は、建設DXサポートシステム構築プロジェクト(18ヶ月)において、プロジェクトに関係するすべてのステークホルダー(経営層、現場監督、IT部門、ベンダー等)に対して、体系的で効果的なコミュニケーションを実施し、プロジェクトの透明性・信頼性を確保することを目的とします。

---

## 2. 適用範囲

**対象**:
- 建設DXサポートシステム(CDCP)構築プロジェクト全期間(Phase 1-3, 18ヶ月)
- プロジェクト関係者全員(内部・外部)

**通信手段**:
- 定例会議(60以上)
- レポート・ドキュメント
- メール・緊急通知
- ダッシュボード・可視化ツール
- 研修・ワークショップ

## 4. 定例会議体

### 4.1 会議体構成

#### **4.1.1 経営会議 (Executive Steering Committee)**

**目的**: プロジェクト全体の意思決定、予算・スケジュール承認、大規模課題エスカレーション

**参加者**: CEO, CFO, 工事部門長, IT部門責任者, PM

**開催頻度**: 月1回(第4火曜 14:00-15:30)

**定例議事**:
1. KPI報告(財務、業務効率、品質、人材)
2. Phase進行状況(スケジュール、予算)
3. 大規模リスク・課題報告
4. 変更要求の承認(Level 3)
5. 予算追加・スケジュール変更検討
6. 次月アクション確認

**配布資料**:
- 月次KPIダッシュボード(1ページ)
- Phase進捗レポート(3-5ページ)
- リスク・課題状況(1ページ)

**議事録作成**: 記録担当(PM)→ 24時間以内に全参加者に配布

**継続時間**: 90分

---

#### **4.1.2 プロジェクト委員会 (Project Steering Committee)**

**目的**: プロジェクト実行の監視・調整、スコープ変更承認(Level 2)、ステークホルダー間調整

**参加者**: PM, 業務責任者(工事部門代表), IT責任者, ベンダーPM, 現場監督代表(2名)

**開催頻度**: 隔週(毎週火曜 10:00-11:30)

**定例議事**:
1. 進捗確認(当週実績、来週計画)
2. 課題・リスク報告と対応状況
3. ユーザーフィードバック共有
4. 変更要求の承認(Level 2)
5. 来週の重要タスク確認
6. アクション項目確認

**配布資料**:
- 週次進捗レポート(2ページ)
- 課題・リスク管理表(1ページ)

**継続時間**: 90分

---

#### **4.1.3 プロジェクト実行会議 (Project Execution Meeting)**

**目的**: 実行チーム(PM, ベンダーPM, IT運用者)による進捗確認、課題解決

**参加者**: PM, ベンダーPM, 開発チームリード, IT運用責任者, 業務分析者

**開催頻度**: 週2回(月曜9:00-9:30, 木曜15:00-15:30) ※スプリント制

**定例議事**:
1. 前スプリント/前週実績報告
2. 当スプリント/当週タスク確認
3. 開発課題・技術的問題解決
4. テスト結果確認
5. 次週タスク割り当て

**配布資料**:
- スプリント進捗ボード(JIRA/Backlog)
- 日次タスク(Slack/Teams通知)

**継続時間**: 30分

---

#### **4.1.4 ユーザーフィードバック会議 (User Feedback Session)**

**目的**: 現場からのユーザー要望・課題を集約、プロダクト改善へ反映

**参加者**: PM, 業務分析者, ベンダーPM, 現場監督代表(2-3名, 月ごと交代), 協力会社代表

**開催頻度**: 月2回(第1水曜・第3水曜 15:00-16:00) ※Phase 1-2

**定例議事**:
1. ユーザーからの新規要望ヒアリング
2. 既知の課題・改善要望の優先度付け
3. 改善案の検討・承認判定
4. 次月以降の改善スケジュール確認

**配布資料**:
- ユーザーフィードバック集約表(1ページ)
- 改善要望の優先度リスト(1ページ)

**継続時間**: 60分

---

#### **4.1.5 ベンダー・パートナーミーティング (Vendor Coordination)**

**目的**: 開発ベンダー・保守ベンダー・他社パートナーとの調整

**参加者(開発ベンダー向け)**:
- ベンダーPM, 開発チームリード, PM, IT責任者

**開催頻度**: 週1回(水曜10:00-11:00)

**定例議事**:
1. 開発進捗確認
2. 技術的課題・解決策
3. 品質・テスト状況
4. リソース・スケジュール調整
5. 納期達成のための対応確認

**配布資料**:
- 開発進捗ダッシュボード
- 技術課題リスト

**継続時間**: 60分

**参加者(保守ベンダー向け)**:
- 保守ベンダーPM, IT責任者, PM

**開催頻度**: 月1回(第2月曜) ※Phase 2以降

**定例議事**:
1. 運用監視状況報告
2. 障害・インシデント報告
3. 性能・容量状況
4. 次月の対応予定

**継続時間**: 45分

---

#### **4.1.6 経営層タウンホール (Executive Town Hall)**

**目的**: 全社員向けにプロジェクト進捗・成果を周知、DX推進への理解・動機付け

**参加者**: 全社員(希望者)、CEO, PM, 工事部門長, 現場監督代表

**開催頻度**: 四半期1回(30分:全社向け + 質問時間30分)

**定例議事**:
1. CEO挨拶(DX推進の重要性)
2. 月間KPI成果報告(15分)
3. 現場事例紹介(現場監督スピーチ 10分)
4. Q&A(質問対応)

**配布資料**:
- 全社向けスライド(20ページ程度)
- 成果事例集(2-3ページ)

**配信方法**: 社内動画ポータルで常時配信(出席できない者向け)

**継続時間**: 60分(内30分はオンサイト、30分はオンライン同時配信)

### 5.2 ダッシュボード・可視化

#### **5.2.1 プロジェクト管理ダッシュボード(リアルタイム)**

**用途**: PM、ベンダーPM、IT責任者が常時アクセス
**ツール**: Backlog, Jira, Trello等のプロジェクト管理ツール
**表示項目**:

| 項目 | 更新頻度 | 表示内容 |
|-----|---------|--------|
| スプリント進捗 | 日次 | 完了/進行中/未開始のタスク割合 |
| バーンダウンチャート | 日次 | 実績vs計画のズレ |
| 不具合管理 | リアルタイム | 新規/対応中/完了の不具合数 |
| リスク・課題 | 日次 | 優先度別のリスク・課題一覧 |

## 7. エスカレーション・課題管理

### 7.1 エスカレーション基準

**Level 1: 即座にPMへ報告**
- 軽微なユーザー質問
- 日常的なタスク進捗
- 小規模なバグ(軽微レベル)

**Level 2: PM → プロジェクト委員会へ報告**
- スケジュール遅延(±1週間)
- 予算超過(±50万円)
- 中程度の技術課題
- ユーザー満足度低下傾向

**Level 3: プロジェクト委員会 → 経営会議へ報告**
- スケジュール大幅遅延(1ヶ月以上)
- 予算大幅超過(200万円以上)
- 高リスク顕在化
- システム稼働率低下
- Phase進行判定への影響

**Level 4: 経営会議 → 取締役会へ報告**
- プロジェクト中止・大幅変更
- 経営戦略への重大影響
- 法的・コンプライアンス課題

### 7.2 課題管理プロセス

**課題の登録**:
1. 課題発見者が「課題レポート」を作成
2. PM承認を得て「課題管理表」に登録
3. 優先度・対応者を割り当て

**課題の追跡**:
- 週次実行会議で課題の状況確認
- 月次レポートで課題集約
- 解決時に完了ステータス更新

**課題の解決**:
| 優先度 | 対応期限 | 対応者 | 報告先 |
|--------|---------|--------|--------|
| 🔴 高 | 3営業日 | PM/ベンダー | プロジェクト委員会 |
| 🟡 中 | 1週間 | ベンダー | プロジェクト委員会 |
| 🟢 低 | 2週間 | ベンダー | 月次レポート |

## 8. クライシス・コミュニケーション

### 8.1 クライシスシナリオ

| シナリオ | トリガー | 対応 |
|---------|---------|------|
| **システムダウン** | 本番稼働中にシステム停止 | 即座に対応チーム招集、CEO/IT責任者に電話連絡、復旧情報をリアルタイム発信 |
| **セキュリティ侵害** | データ流出、不正アクセス検知 | 法務・IT責任者と対応、ステークホルダーへの報告、対外発表検討 |
| **大規模スケジュール遅延** | Phase完了が1ヶ月以上遅延 | 経営会議で原因分析、対応策提示、関係ベンダーへの対応 |
| **予算大幅超過** | 追加費用が予備費超過 | 経営会議で新規予算審議、ステークホルダー説明 |
| **主要ベンダー撤退** | ベンダー側リソース削減・撤退 | PM～CEO連携で代替ベンダー手配、スケジュール再調整 |

### 8.2 クライシス対応時の通信

**基本原則**: 正直・透明・迅速

**対応チーム**: PM + CEO + IT責任者 + 工事部門長

**対応フロー**:
1. **初期対応**(30分以内)
   - 事実確認
   - CEO/工事部門長に電話報告
   - 対応チーム招集

2. **情報発信**(1時間以内)
   - 全ステークホルダーへメール: 状況説明 + 対応方針
   - 内部向けメール: "進行中、詳細は随時お知らせ"

3. **詳細対応**(24時間以内)
   - 原因分析報告書
   - 対策案
   - 今後の予定

4. **フォローアップ**(継続)
   - 復旧/解決までの日次報告
   - 完了後の報告書作成
